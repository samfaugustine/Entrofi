01  Foundation

Install and authenticate

Pick your OS for the full path — install, first run, log in, verify. The native installer is recommended everywhere: it auto-updates and needs no Node.js. Other methods (Homebrew, WinGet, package managers, npm) sit under "other install methods."

Needs: macOS 13+ · Apple Silicon or Intel · 4 GB RAM. Node.js 18+ only if you pick the npm path.

1

Install (native installer)

curl -fsSL https://claude.ai/install.sh | bash

Installs to ~/.local/bin/claude and updates itself in the background.

Other install methods

Homebrew (manual upgrades):

brew install --cask claude-code

npm — needs Node 18+ (brew install node first):

npm install -g @anthropic-ai/claude-code
2

Start it and log in

cd ~/your-project
claude          # launches the session
/login          # opens the browser to authenticate

You need a Claude Pro, Max, Team, or Enterprise plan, or an Anthropic Console (API) account — the free plan doesn't include Claude Code. If the browser doesn't open, press c to copy the login URL.

claude doctor   # checks install, config, dependencies

Add ~/.local/bin to your PATH, then reload your shell:

03  The highest-leverage move

Give it a memory — CLAUDE.md

This is the difference between a generic assistant and one that knows your project. CLAUDE.md is plain markdown that Claude Code loads automatically at the start of every session. Put your conventions, architecture notes, "don't do X," and commands in it.

/init   # scans your project and drafts a CLAUDE.md for you

There's a hierarchy, and the files compose — they all load together, most-specific wins:

LocationScope
~/.claude/CLAUDE.mdYou, everywhere — global preferences across all projects
<repo>/CLAUDE.mdThis project — checked into git, shared with the team
<repo>/subdir/CLAUDE.mdA subfolder — loads when you work in that part of the tree
CLAUDE.local.mdYour personal, un-committed overrides for this repo

You can @import other files into a CLAUDE.md (e.g. @docs/architecture.md) to keep the main file lean while pulling in deeper context on demand. Manage it with /memory.

The discipline that pays off

Treat CLAUDE.md as a living file. Every time Claude gets something wrong because it lacked context, add the missing fact to the file instead of re-explaining it next session. It becomes your project's institutional memory.

04  The biggest multiplier

Connect your world — MCP servers

MCP (Model Context Protocol) is the plugin system. Each server gives Claude Code a new set of tools — read your database, search your docs, manage GitHub issues, post to Slack, query a knowledge vault.

claude mcp add <name> -- <command to run the server>
claude mcp list
/mcp                # check connection status mid-session

Servers scope to local (just you), project (committed via .mcp.json so your whole team gets them), or user (all your projects). Transports are stdio, SSE, or HTTP.

Obsidian as the persistent knowledge layer

This is the pattern worth copying. Obsidian is a local-first markdown notes app; pointing an MCP server at your vault gives Claude Code a durable, searchable knowledge base that lives outside any single conversation.

claude mcp add-json obsidian --scope user \
  '{"type":"stdio","command":"npx",
    "args":["-y","@bitbonsai/mcpvault@latest","/path/to/YourVault"]}'

The server is pure file-based — it doesn't even need the Obsidian app running, just the folder of markdown files (Node ≥ 18). Why it's powerful:

  • Memory that survives /clear and new sessions. Conversations are ephemeral; a vault note is forever. Claude writes findings to the vault and reads them back weeks later.
  • A second brain you both edit. You write notes normally; Claude searches them, cites them with clickable links, and adds new notes in your conventions.
  • Linking builds a knowledge graph. Obsidian's [[wiki-links]] let Claude connect related notes and synthesize across sources — not just retrieve one file.
The general principle

Give Claude a place to put durable knowledge, and a CLAUDE.md that tells it when to read and write there. That turns one-off help into compounding expertise.

05  Package what repeats

Make it repeatable

Once you notice yourself typing the same instructions repeatedly, package them.

Custom slash commands

A markdown file becomes a command. Drop review-pr.md in .claude/commands/ and it becomes /review-pr. Use $ARGUMENTS, $1, $2 for parameters. You explicitly run a saved prompt.

Skills

A folder with a SKILL.md that Claude loads automatically when the task matches. Progressive disclosure keeps it cheap until needed. Claude auto-invokes a packaged capability.

Hooks

Shell commands the harness runs on events — PreToolUse, PostToolUse, Stop, SessionStart — set in settings.json. Deterministic enforcement Claude can't forget.

The distinction that trips people up: a slash command is a prompt you run; a skill is a capability Claude invokes for you; a subagent is a separate Claude entirely. Hooks fire deterministically — that's their whole value. Use them for non-negotiables like "run the formatter after every edit" or "block edits to prod.env."

06  When one conversation isn't enough

Scale yourself

Subagents are separate Claude instances with their own context window, system prompt, tools, and model — markdown files in .claude/agents/, managed with /agents. Two reasons they're powerful:

  • Context isolation — a subagent can read 40 files and hand back a 3-line conclusion, keeping your main session clean.
  • Specialization + parallelism — run a code-reviewer, a test-writer, and an explorer concurrently, each with its own instructions.

Multi-agent workflows are the newest tier: Claude orchestrates many subagents in the background against a deterministic script — fan out across files, verify findings adversarially, synthesize. Genuinely advanced and token-hungry; reach for it on large migrations, audits, or comprehensive reviews, not daily edits. Get fluent with single subagents first.

Headless mode ties it to automation: claude -p "your prompt" runs non-interactively, and --output-format json makes it scriptable. Combined with the GitHub app (@claude mentions on PRs trigger it in CI), Claude Code becomes part of your pipeline, not just your terminal.

Don't skip these

Round out your setup

Beyond the core loop, these six round out a setup you can lean on day to day. Three of them — your first real session, the IDE, and safety — are worth reading in week one.

The fastest way to get the feel — 15 minutes in a repo you already know.

1

Point it at a real project

cd ~/some-project
claude

Then ask it to orient itself: "Explain what this codebase does and how it's structured." Watch it read files before answering — that's the agent loop in action.

2

Make one small change in plan mode

Type /plan, then ask for something concrete: "Add input validation to the signup form." It proposes a plan and waits — you approve before it touches a file.

3

Review the diff, then decide

It shows every change. Accept it, ask for revisions, or /rewind to undo. Nothing ships without you seeing it.

4

Capture what it got wrong

If it missed a convention, tell it — then add that fact to CLAUDE.md so it's right next time.

The habit to build

Small, reviewable changes in plan mode beat one giant ask. You stay in the loop; it stays accurate.

The terminal is the clearest place to start, but the VS Code and JetBrains extensions add inline diffs and selection context. Both still need the standalone CLI installed — they don't replace it.

In VS Code

Install the Claude Code extension from the Marketplace (or open vscode:extension/anthropic.claude-code). Launch it from the ✱ Spark icon in the editor toolbar or sidebar, or Cmd/Ctrl+Shift+P → "Claude Code". You get a side-by-side diff view, @-file mentions with line ranges, and plan review inside the editor.

In JetBrains

Install Claude Code [Beta] from the JetBrains Marketplace and restart the IDE. Open it with Cmd+Esc (Mac) or Ctrl+Esc (Windows/Linux). It shares your current selection and IDE diagnostics automatically.

Already in a terminal session?

Run /ide to connect a running terminal session to your open editor — you get the diff view and diagnostics without restarting.

Claude Code is multimodal — hand it screenshots, mockups, diagrams, or PDFs, not just text.

  • Drag & drop an image into the VS Code extension or the desktop app.
  • Paste from the clipboard — Ctrl+V in most terminals (Cmd+V in iTerm2). Perfect for a quick screenshot of an error or a broken UI.
  • Reference a file path in your prompt: "Match this layout: ./design/home.png".

What it's good for

  • Paste a screenshot of a broken screen and ask it to fix the CSS.
  • Drop in a design export and ask for the component.
  • Hand it a PDF spec and ask it to implement against it.

The file pays off in proportion to what you put in it. Aim for what you'd tell a new teammate on day one — concrete commands, where things live, and the rules that aren't obvious from the code.

# CLAUDE.md

## Commands
- Test: `npm run test`
- Lint: `npm run lint` (run before every commit)
- Dev server: `npm run dev`

## Architecture
- `src/api/` — REST handlers · `src/db/` — Prisma models
- Auth lives in `src/auth/`; never roll your own.

## Conventions
- TypeScript strict mode. No `any`.
- Use the existing `Result<T>` type for fallible calls.

## Don't
- Don't edit `generated/` — it's built from the schema.
- Don't add dependencies without asking first.

@docs/architecture.md   # pull in deeper context on demand
Keep it lean

It loads into every session's context, so link out with @import instead of pasting long docs inline. Grow it when Claude gets something wrong — don't pre-write a novel.

Using it heavily and sustainably comes down to two habits: pick the right model, and watch your context window.

Right model for the job

Opus or Fable for hard reasoning, Sonnet for routine edits, Haiku for mechanical work. Switch with /model. Running the biggest model on small tasks is the most common way to overspend.

Watch usage and context

  • /usage — your plan, session and weekly usage bars, and which skills, subagents, and MCP servers ate the most.
  • /context — what's filling the context window (history, files, CLAUDE.md, loaded skills). A bloated context costs more and degrades quality.
  • /clear between unrelated tasks; /compact (or /compact focus on the API changes) to summarize a long one.
Subscription vs API

A flat Pro/Max plan is best for heavy interactive use; a pay-per-token API key suits spiky usage or scripting into CI.

Speed is the point, but a few defaults keep you out of trouble. This is the judgment that separates people who use it well from people who get burned.

Permission modes + your allowlist

By default Claude asks before edits and commands. acceptEdits stops asking for file edits, plan is read-only-propose, and bypassPermissions skips all prompts. Tune what's auto-allowed in settings.json — deny always wins over allow:

{
  "permissions": {
    "allow": ["Bash(npm run lint)", "Read(~/.zshrc)"],
    "deny":  ["Read(./.env)", "Read(./secrets/**)"]
  }
}

OS-level sandboxing

For isolation of bash commands, /sandbox (or "sandbox": {"enabled": true} in settings) confines filesystem and network access. Built in on macOS (Seatbelt) and Linux/WSL2 (needs bubblewrap + socat); not on native Windows. Safe commands can then run without prompts, because the OS enforces the boundary.

Bypass mode is for sandboxes only

--dangerously-skip-permissions (the bypassPermissions mode) skips every prompt — only use it inside a container or VM where Claude can't do harm. Even then, rm -rf / and rm -rf ~ stay blocked as a circuit breaker.

MCP servers can read your data

A connected server gets real access — your database, files, messages. Only add servers you trust, and check what each one is allowed to do.

Always review the output

Claude is fast and usually right, not infallible. Read the diff before accepting, run the tests, and don't ship generated code you haven't understood. Plan mode plus small changes makes this the easy path, not the chore.

Don't do it all at once

A sane learning path

Trying every layer in week one is how people bounce off it. This order compounds.

  1. 1
    Week 1 — live in the loop
    Install, then use /plan on anything non-trivial, @-mention files, /clear between tasks. Get the rhythm.
  2. 2
    Week 1–2 — grow your CLAUDE.md
    Run /init, then add a fact every time Claude lacks context. This alone doubles its usefulness.
  3. 3
    Week 2–3 — wire in MCP
    Add 1–2 servers you'll actually use. If you keep notes, connect Obsidian as durable memory.
  4. 4
    Week 3+ — package the repeats
    Turn recurring prompts into slash commands and skills. Add a hook or two for your non-negotiables.
  5. 5
    At the ceiling — scale
    Reach for subagents, then multi-agent workflows, when one conversation can't hold the job.
